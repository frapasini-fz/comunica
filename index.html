ComUnica

Mappa dei moduli che compongono una Comunicazione Unica di Impresa — filtrata allo scope "impresa individuale non agricola". Sono escluse tutte le forme societarie (SP/SC/Consorzio/GEIE/EPE/REA/Trust) e tutte le modulistiche agricole (INPS DA, CD, IAP, INAIL QuadroP, AdE AA7/10).

1. Struttura della pratica e nomi dei file

Ogni comunicazione unica d'impresa è una collezione di file, strutturata in sezioni dedicate a ciascun ente destinatario (spec ComUnica v3.1). Ogni pratica informatica ha un codice pratica di 8 caratteri abbcddee che codifica la data-ora di costruzione:

CodiceSignificatoValori
aMese1=gennaio, 2=febbraio, ..., 9=settembre, A=ottobre, B=novembre, C=dicembre
bbGiorno01–31
cOraA=ore 1, B=ore 2, ..., I=ore 9, J=ore 10, ..., W=ore 23, X=ore 24 o 00
ddMinuto00–59
eeSecondo00–59
Esempio: il codice per una pratica preparata il 12 Novembre alle ore 16:28:37 è B12P2837 (B=Novembre, 12=giorno, P=ora 16, 28=minuto, 37=secondo). È il codice di esempio usato in tutto il documento e nel diagramma.

7. Modulistica SUAP — SCIA Contestuale via SSU

Dal nuovo flusso normato dal D.P.R. 160/2010 e dal Sistema SSU (Sistema informativo degli Sportelli Unici, art. 3 dell'Allegato tecnico al D.P.R. 160/2010), Fiscozen non assembla più lo ZIP della pratica SUAP. La SCIA è compilata sul portale del Comune; Fiscozen riceve solo un CUI uuid (Codice Univoco di Istanza) da agganciare al file B12P2837.CUI.SUAP.xml della ComUnica.

Riferimento normativo e tecnico: Manuale Operativo — Servizio integrazione applicativa Comunicazione Unica con SCIA contestuale (32 pag., InfoCamere). I riferimenti puntuali a §X / pag. Y sono indicati accanto a ciascun passo.

7.1 Flusso end-to-end

  1. Fiscozen genera il codice pratica ComUnica (es. B12P2837) secondo Spec ComUnica v3.1. Manuale Op. §4.1.1 (pag. 9)
  2. Fiscozen richiede a InfoCamere l'URL del portale SUAP del Comune competente, passando: codice pratica ComUnica, codice ISTAT del Comune, indirizzo del servizio di notifica. §4.1.1, §4.1.2 (pag. 9-11)
  3. InfoCamere risponde con l'URL del portale SUAP. Se il Comune non è ancora attivato sulle nuove specifiche SSU → HTTP 503 "municipality not active" → fallback PEC manuale (vedi §7.5). §4.1.2 (pag. 11)
  4. Fiscozen invia il consulente al portale. L'utente che compila la SCIA può essere diverso da quello che usa Fiscozen; il primo che apre l'URL diventa "proprietario" della SCIA sul portale del Comune.
  5. Il consulente compila la SCIA direttamente sul portale del Comune. Fiscozen non vede i campi della SCIA — il portale è del Comune, non nostro.
  6. Il cliente paga i diritti SUAP via PagoPA contestualmente alla compilazione. La ricevuta finisce automaticamente nel general_index come RICEVUTA_PDF. §5.1 (pag. 23-24)
  7. Il portale SUAP notifica Fiscozen via suap_send_instance con il CUI uuid + l'indice file general_index (struttura JSON che cataloga tutti gli allegati assemblati dal Comune). §5, §5.2 (pag. 21, 30)
  8. Fiscozen aggancia il CUI dentro il file B12P2837.CUI.SUAP.xml della ComUnica. §4.1, §4.2.1 (pag. 9, 12)
  9. Fiscozen invia la ComUnica al RI via Telemaco. Il RI inoltra al SUAP comunale che chiude la pratica.

Appendice A — Tracciato FEDRA (come si compila la pratica RI)

I "moduli" RI elencati sopra (I1, I2, S5, UL, IntercalareP, R, ...) non sono file XML autonomi: sono insiemi logici di campi che vengono serializzati in un unico file di testo binario record-based — il file-pratica FEDRA. Il tracciato è standardizzato dal DM MIMIT 12/04/2023 (Allegato A v7.0+) e generato esclusivamente da software autorizzati (StarWeb, FEDRA, ComUnica, software dei commercialisti): non si scrive a mano.

I tre artefatti che la ComUnica trasporta per il RI

FileEstensioneContenutoGenerato da
file-pratica .U3T Dati strutturati FEDRA (record a coordinate X Y Z) — il "contenitore dati" che il RI legge con parser automatico Software compilatore
file-distinta .PDF.P7M Visura a quadri leggibile, firmata digitalmente CAdES — documento legale di sottoscrizione Software compilatore + firma dichiarante/intermediario
file-riepilogo .U3R Record fissi a 320 byte: catalogo metadati di ogni documento allegato (nome, tipo da TABELLA_DOC, pagine, data, codice atto) Software compilatore

Il file-distinta firmato è il documento "legale"; il file-pratica e il file-riepilogo sono i dati machine-readable che il sistema RI elabora.

Coordinate dei record: (X, Y, Z)

Ogni campo valorizzato della modulistica produce un record identificato da una tripla:

Coord.ByteSignificatoEsempi
X 1 char Identificativo modulo 1=I1, 3=I2, 5=UL, P=IntercalareP, O=RP (riepilogo) — moduli applicabili alla Ditta Individuale
Y 1 char Identificativo riquadro/sezione dentro il modulo 1=Dati anagrafici, 2=Residenza, 85=Tipo domanda, ...
Z 2 byte (unsigned short) Identificativo campo all'interno della sezione 10=CF, 20=P.IVA, 40=cognome, 50=nome, ...
Il primo record di ogni modulo ha sempre Y=0, Z=1 con contenuto fisso di 6 byte ASCII '0' (0x30) — fa da marcatore di inizio modulo. I moduli ripetibili (IntercalareP per ogni socio/familiare, UL per ogni unità locale) ricominciano da capo questa sequenza per ogni occorrenza.

Tipi di record nel file-pratica

  1. Record di testa (HEADER) — 8 byte fissi: offset al record di coda + filler 0x68 0x01 0x00 0x00. Una sola volta a inizio file.
  2. Record dati (DATI MODULISTICI) — uno per ogni campo: X (1) | Y (1) | Z (2 bytes) | filler 0x00 0x00 | contenuto (n byte) | 0x7C (pipe). Si ripete per N campi della pratica.
  3. Record di coda (FOOTER) — 360 byte fissi: filler 0x05 + CF impresa (16) + Identificativo source software (12) + versione (3) + REA + ModuloBase + denominazione pratica (84) + sequenza enti destinatari + sequenza moduli (60) + userid utente. Definisce il routing della pratica verso AdE/INPS/INAIL.
  4. Header di file (80 byte, aggiunto dopo formattazione finale e "targatura") — numero totale record + sigla provincia destinataria. È il metadato di trasmissione telematica.

Dopo la generazione, il file viene formattato in blocchi di 80 byte (operazione 6.1 dell'Allegato A) e poi "targato" con l'header di file (operazione 6.2).

Flusso end-to-end di compilazione

  1. L'utente compila i campi sui moduli logici (I1, I2, IntercalareP, RP, ...) tramite l'interfaccia del software (StarWeb, FEDRA, ComUnica, software di studio).
  2. Il software serializza ogni campo nel relativo record X Y Z + contenuto + 0x7C.
  3. Aggiunge i metadati nel FOOTER: identificativo source, versione ("700" per v7.0), sequenza enti destinatari (AE/IL/IP/RI/AA/ML/SP), sequenza moduli compilati, userid.
  4. Formatta il file in blocchi da 80 byte e targa con l'header di routing (numero record + sigla provincia).
  5. Genera in parallelo il file-distinta (PDF stampabile della pratica) e il file-riepilogo (U3R).
  6. La distinta viene firmata digitalmente dal dichiarante o dall'intermediario (CAdES).
  7. Il software impacchetta tutto nella ComUnica: il ModelloBase referenzia file-pratica, file-distinta e file-riepilogo come allegati con la rispettiva impronta digitale (vedi §1). Il ModelloBase aggiunge poi gli analoghi file-pratica/file-distinta degli altri enti destinatari (AdE, INAIL, INPS, SUAP).
  8. Trasmissione telematica via Telemaco/portale ComUnica.
Implicazione progettuale: i nomi "logici" usati nella documentazione interna ("modulo I1", "riquadro AC", "campo cognome") sono diversi dai nomi reali del wire-format. Chi implementa un generatore di pratiche RI deve scrivere uno strato di serializzazione FEDRA che traduce dal modello logico (es. Persona.cognome) alle coordinate fisiche (1 1 40 ROSSI|). Gli enti non RI (AdE, INAIL, INPS, SUAP) hanno invece tracciati XML/posizionali distinti — vedi §4–§7.
